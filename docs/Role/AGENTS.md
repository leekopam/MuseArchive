# Project Coding Rules

1. MVVM 패턴과 OOP를 적극적으로 사용해 각 기능들을 구현
2. 결합도는 낮추고 응집도는 높이는 형태로 코드를 작성해 모듈 간 의존성을 줄여 유지보수성과 재사용성을 극대화
3. 필요 이상의 수정은 자제하며 안정성을 우선시하며 원래 정상적으로 작동하던 기능들이 코드를 작성하는 과정에서 절대로 문제가 발생하는 경우는 없어야 한다
4. 코드는 클린코드 원칙을 따르며 최대한 간결하고 단순하게 작성하여 신뢰성과 유지보수를 하기 쉽도록 작성, 코드는무작정 짧게 구현하는것이 아니라 구현하는 기능이 정상적으로 작동해야한다는 전제를 깔고 지켜져야한다
5. 10년차 이상의 시니어 개발자가 됐다고 가정하고 전문적인 지식과 전문 기술을 모두 최대치로 사용해 문제를 해결하고자 할때 지금 문제를 가장 효과적으로 해결 할 수 있는 방법을 찾아내서 논문급으로 아주 자세하게 작업지시서 제작한뒤 승인받은 경우에만 작업을 시작 단 바로 작업을 시작하라고 하는 경우는 예외
6. 주석은 무슨 기능인지 알아볼 수 있정도로 간결하고 가독성 좋게 작성, 반드시 한국어로 작성해야 한다
7. 자료구조, 알고리즘을 적극적으로 사용해 데이터를 체계적으로 조직하고 효율적으로 사용할 수 있도록 코드를 작성
8. 주석이나 UI는 반드시 전부다 "한국어"로 작성하며 필요에 따라 영어를 일부 허용한다 하지만 이모티콘 사용은 금지한다.

## Engineering Quality Rules (MUST)

> 목적: "잘 동작하는 코드"를 넘어서, **회귀/장애/디버깅 불가/성능 저하**를 사전에 막기 위한 공통 품질 규칙.
> - UI/UX 관련 상세 규칙은 `docs/SKILL/uiux.md`에서 다룬다.

E1. **변경 의도 1개 원칙**: 한 번의 작업/PR/커밋은 하나의 목적(버그 수정/기능 추가/리팩터링/성능 개선)을 유지한다. 기능 변경과 리팩터링은 가능하면 분리한다.

E2. **기존 컨벤션 최우선**: 프로젝트의 디렉토리 구조, 네이밍, 레이어 분리, DI/상태 관리/에러 처리 방식 등 기존 패턴을 최우선으로 따른다. 새로운 패턴/프레임워크 도입은 근거·대안·마이그레이션 계획이 있을 때만 허용한다.

E3. **명시적 에러 처리**: 예외/에러를 catch 후 무시하지 않는다. (1) 상위로 전파, (2) 사용자/호출자가 재시도 가능한 형태로 변환(Result/Either 등), (3) 안전한 대체 경로 제공 중 하나를 선택한다. 필요한 경우 타임아웃/재시도(백오프) 정책을 포함한다.

E4. **경계에서 검증**: 외부 입력(네트워크/스토리지/유저 입력/딥링크/환경변수)은 경계에서 검증·정규화한 뒤 내부로 전달한다. 신뢰 경계 밖 데이터는 항상 불완전하다고 가정한다.

E5. **타입 안정/불변성 우선**: Nullable/동적 타입/Any 남발을 금지한다. 가능한 범위에서 불변(immutable) 데이터 구조를 우선하며, 사이드이펙트(I/O)는 경계로 격리한다.

E6. **테스트 동반 원칙**: 신규 로직/버그 수정에는 최소 1개 이상의 테스트를 추가한다.
- 버그 수정: 재현 테스트(회귀) 우선
- 신규 로직: 핵심 비즈니스 로직 단위 테스트 우선
- 상태가 있는 기능: 상태 전이 테스트 우선

E7. **품질 게이트**: 제출 전 기준을 만족해야 한다.
- 빌드/타입체크/린트/포맷/테스트가 실패하거나 경고가 누적되면 제출하지 않는다.
- 실제 실행이 불가능한 환경이라면, 최소한 실행 커맨드/예상 결과/검증 포인트를 명시한다.

E8. **의존성 최소화**: 새로운 라이브러리는 "표준 라이브러리/기존 의존성"으로 해결이 불가능할 때만 추가한다. 도입 시 이유, 대안, 영향(바이너리 크기/성능/보안/라이선스)을 함께 남긴다.

E9. **관측 가능성(Observability)**: 장애 분석에 필요한 최소 로그/에러 코드/컨텍스트를 남긴다. 단, 개인정보/민감정보는 로그·분석 이벤트에 절대 포함하지 않으며 필요 시 마스킹한다.

E10. **배포/롤백 안전**: 스키마/설정/캐시/프로토콜 변경은 backward-compatible를 우선한다. 마이그레이션/롤백 전략이 없으면 위험한 변경을 진행하지 않는다.

## Skill Routing (필요할 때만 추가 적용)

> 목적: `AGENTS.md`는 **항상 적용되는 코어 규칙**만 유지하고,  
> 작업 성격에 따라 **Skill 파일을 추가로 첨부**해 컨텍스트를 최소화한다(정확도/작업 효율 ↑).

### UI/UX 작업(화면/컴포넌트/폼/내비게이션)
- 적용 Skill: `docs/SKILL/uiux.md`
- (권장) 요구사항 명세 템플릿: `docs/UI/uiux_request.md` (값을 채워서 제공)

---

## 적용 우선순위(충돌 시)

1) 사용자가 이번 작업에서 명시한 요구사항/스펙(티켓/PRD/Figma 포함)  
2) `AGENTS.md` 코어 규칙(MUST)  
3) 추가로 적용한 Skill 규칙(`skills/*.md`)  
4) 프로젝트/프레임워크 컨벤션(기존 코드 스타일, 린트/포맷터, 아키텍처)  
5) 일반적인 베스트 프랙티스

---

## (권장) 모델 응답 형식(Output Contract)

> 아래 형식으로 답변하면 누락이 크게 줄어든다.
> - 사용자가 "바로 코드"를 요구해도, 최소한 1)~2)는 짧게라도 포함한다.

1) 해석/가정(Assumptions)  
2) 작업 계획(변경 파일/구조/핵심 로직/에러 처리/테스트 전략)  
3) 구현(코드)  
4) 테스트(추가/수정)  
5) 수동 QA 체크리스트(재현 단계 포함)  
6) 리스크/롤백 포인트

- 일반 작업(로직/리팩터링/버그 수정)
  - "첨부한 `AGENTS.md` 규칙을 준수해."

- UI 작업(화면/컴포넌트/폼/내비게이션)
  - "첨부한 `AGENTS.md` 규칙 + `docs/SKILL/uiux.md` 규칙을 함께 준수해."
