# Skill: UI/UX Implementation (MUST)

> 목적
> - UI 작업에서 **상태 누락(Loading/Empty/Error/Disabled)**, **접근성 미준수**, **일관성 붕괴**, **체감 성능 저하** 같은 품질 이슈를 사전에 차단한다.
> - 본 Skill은 "UI/UX 구현 규칙"과 "UI 작업 시 모델 응답 산출물(Output Contract)"만 포함한다.
> - (권장) 요구사항 명세는 별도 템플릿 파일(`UIUX_REQUEST_TEMPLATE.md`)을 채워서 제공한다.

---

## 적용 우선순위(충돌 시)

1) 이번 작업의 명시 요구사항/스펙(티켓/PRD/Figma/기존 화면 정책 포함)
2) 본 Skill의 UI/UX 규칙(MUST)
3) 프로젝트/프레임워크 컨벤션(기존 코드 스타일, 린트/포맷터, 아키텍처)
4) 일반적인 베스트 프랙티스

> 참고: `AGENTS.md`의 Project Coding Rules(예: MVVM/OOP, 안정성, 주석, 알고리즘 등)은 UI 작업에서도 **항상** 적용된다.

---

## When to use (이 Skill을 붙이는 경우)

- 화면/컴포넌트 구현 또는 수정
- 레이아웃/반응형/다크모드/텍스트 확대 대응
- 폼(입력/검증) UX
- 내비게이션/인터랙션(제스처/뒤로가기 포함)
- 디자인 시스템 적용/리팩터링
- UI 성능 개선(리스트/이미지/애니메이션)

## UI/UX Coding Rules (MUST)

8. UI/UX 구현은 "사용자 목표(Why) → 핵심 시나리오(Flow) → 성공 기준(DoD)"을 먼저 정의하고, 정의된 흐름을 절대 깨지 않도록 구현한다  
   - (체크) 기존 플로우가 있으면 **회귀 금지**가 우선이다.

9. 디자인 시스템/기존 컴포넌트/토큰(색상, 폰트, 간격, radius 등)을 최우선으로 사용하고, 임의의 매직 넘버/인라인 스타일 하드코딩을 금지한다(필요 시 토큰/스타일로 승격)  
   - (체크) 신규 스타일이 필요하면 **토큰/공용 스타일로 승격**하고 이유를 남긴다.

10. 모든 화면/컴포넌트는 최소한 Loading / Success / Empty / Error / Disabled 상태를 명시적으로 모델링하고, 각 상태에서의 UI·인터랙션·문구를 구현한다(네트워크 지연/오프라인 포함)  
   - (체크) "무한 스피너" 금지. Error/Empty에는 **다음 행동(CTA/재시도/가이드)** 이 있어야 한다.

11. 접근성(A11y)을 기본값으로 준수한다: 스크린리더 라벨/역할/순서, 키보드/리모컨 포커스 이동, 터치 타겟 크기, 명도 대비, 동적 폰트(텍스트 확대), 다크모드, Reduce Motion(과도한 애니메이션 회피)  
   - (체크) 입력/버튼/아이콘은 **의미 있는 라벨**을 가진다. 포커스 이동이 자연스럽고 포커스 표시가 명확해야 한다.

12. 반응형/적응형 레이아웃을 적용해 다양한 화면 크기/해상도/회전/안전영역(Safe Area)에서 레이아웃 깨짐이 없도록 구현한다(고정 폭/고정 높이 최소화)  
   - (체크) 긴 텍스트/다국어 확장 시 줄바꿈/오버플로우 대응.

13. 입력 UX는 "오류 예방"을 최우선으로 한다: 실시간/단계적 검증, 인라인 에러 표시, 적절한 키보드 타입/입력 마스크/자동완성, 포커스 이동, 제출 전 중복 클릭 방지  
   - (체크) 검증 실패 시 **어디가 문제인지/어떻게 고치는지** 즉시 알 수 있어야 한다.

14. 사용자 피드백은 즉시·일관되게 제공한다: 로딩 인디케이터/스켈레톤, 성공/실패 토스트·스낵바, 진행률 표시, 재시도 버튼, 파괴적 액션은 확인/Undo(가능한 경우) 제공  
   - (체크) 사용자가 눌렀는데 아무 반응이 없으면 실패다(즉시 피드백 필수).

15. 사용자에게 노출되는 모든 문구(버튼/안내/에러)는 간결하고 자연스러운 한국어로 작성하고, 문자열 리소스로 분리해 i18n(다국어)/포맷(날짜·숫자·통화)에 대응 가능하게 만든다  
   - (체크) 에러 문구는 "문제 + (가능하면) 원인 + 해결 방법" 형태를 따른다.

16. UI 성능을 제품 품질로 취급한다: 스크롤/애니메이션 끊김(jank) 방지, 큰 리스트 가상화/페이징, 이미지 지연 로딩/캐싱, 메인 스레드 블로킹 금지, 불필요한 리렌더/리컴포지션 최소화  
   - (체크) 리스트/그리드/피드는 **가상화/페이징** 전제를 기본으로 검토한다.

17. 내비게이션/인터랙션은 일관성을 최우선으로 하며, 기존 사용자 습관을 깨는 변경(버튼 위치, 제스처, 뒤로가기 동작 등)은 명시적인 요구사항이 없는 한 금지한다  
   - (체크) Back 동작/닫기 동작/완료 후 이동 규칙이 일관되어야 한다.

18. UI/UX 변경은 회귀(regression) 방지를 필수로 한다: 주요 화면/흐름에 대한 UI 테스트(또는 스냅샷 테스트), ViewModel 단위 테스트, 접근성 체크를 포함해 "변경 전/후" 동작 동일성을 검증한다  
   - (체크) 최소한 핵심 플로우는 수동 QA 체크리스트로라도 검증 단계를 남긴다.

---

## 최소 요구사항 입력(Quick Spec, 템플릿 대신 급하게 쓸 때)

아래 항목이 제공되지 않으면, 임의로 추측하지 말고 **Assumptions(가정)** 을 명시해 구현한다.

- 사용자 목표(Why) 1문장
- 핵심 플로우(Flow) 1~3단계
- 성공 기준(DoD): 사용자가 무엇을 하면 “성공”인지
- 상태: Loading/Empty/Error/Disabled에서 보여줄 UI + CTA
- 주요 액션(Primary CTA) / 보조 액션(Secondary)
- 접근성 요구(최소: 라벨/포커스/터치 타겟/텍스트 확대)
- 반응형 범위(최소 화면/최대 화면, 회전 여부)
- 성능 요구(리스트 규모, 이미지 여부)
- 회귀 금지(절대 깨지면 안 되는 기존 동작)
- 테스트/QA 범위(최소 수동 QA 시나리오)

---

## Output Contract (UI 작업 시 모델이 반드시 포함해야 하는 산출물)

> 사용자가 “바로 코드부터 작성”을 명시하지 않았다면, `AGENTS.md` 규칙에 따라 먼저 **상세 작업지시서(설계/리스크/테스트 포함)** 를 작성하고 승인 후 구현한다.

1) 해석/가정(Assumptions)  
2) 화면/플로우 요약(Why → Flow → DoD)  
3) 상태 모델 정의(Loading/Success/Empty/Error/Disabled + 이벤트/액션)  
4) 컴포넌트 구조/책임 분리(View ↔ ViewModel ↔ UseCase/Repository)  
5) 접근성 체크리스트(라벨/포커스/대비/텍스트 확대/Reduce Motion)  
6) 반응형/다크모드/긴 텍스트 대응 포인트  
7) 성능 고려(리스트/이미지/리렌더/메인스레드)  
8) 구현(코드)  
9) 테스트(단위/UI/스냅샷 중 가능한 범위)  
10) 수동 QA 체크리스트(재현 단계 + 기대 결과)  
11) 리스크/롤백 포인트

---

## DoD 셀프체크(제출 전)

- [ ] Loading/Empty/Error/Disabled가 모두 동작하고, 각 상태에 사용자 액션(CTA)이 있다
- [ ] 접근성: 라벨/포커스/키보드(가능한 경우)/터치 타겟/대비/텍스트 확대를 고려했다
- [ ] 반응형: 작은 화면/큰 화면/회전/안전영역에서 깨지지 않는다
- [ ] 문구가 자연스럽고(한국어), 리소스 분리(i18n)가 가능하다
- [ ] 스크롤/애니메이션이 끊기지 않도록 성능을 고려했다
- [ ] 기존 동작 회귀가 없다(테스트 또는 QA 체크리스트로 검증했다)
